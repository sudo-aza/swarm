**Action**: v3.28 — fixed everypar re-injection for multi-paragraph narrowing. Task #161 resolved.

**Context**: VM fully reset — re-cloned repo, re-ran setup.sh, texhash, installed missing packages (lipsum, csquotes, etc.). Pulled latest (deca402, QA 09:30 turn with v3.27 recompile and 1420 overlap detection). Found task #161 pending — QA confirmed 1420 body-text overlaps on 107 pages where paragraph 2+ flows at full width through figures.

**Root cause analysis (task #161)**: The v3.25 multi-paragraph extension (`\everypar{\swarmwrap@set@parshape}` + remaining counter) had a critical flaw. TeX's `\everypar` mechanism works as follows: when a paragraph starts, TeX expands the `\everypar` token list, then CLEARS it to empty. The v3.25 code set `\everypar` once in `\swarmwrapnext` (line 553), which fired for the FIRST paragraph (via `\noindent` on line 554). But `\swarmwrap@set@parshape` did NOT re-set `\everypar` for subsequent paragraphs. So when `\lipsum[2]` started a new paragraph (after `\par` from `\lipsum[1]`), `\everypar` was empty — no parshape was injected — text flowed at full width through the figure.

Additionally, the Lua code updated `tex.count[286]` (hardcoded register number) but `\swarmwrap@remaining` may not be register 286 depending on what other packages loaded. The TeX macros read `\swarmwrap@remaining` (wrong register), while Lua updated a different register.

**Fix** (three parts):
1. `\swarmwrap@set@parshape` now re-sets `\everypar{\swarmwrap@set@parshape}` when `remaining > 0`. This chains the narrowing across ALL paragraphs until the narrow zone is exhausted. This is the critical fix for the 1420 overlaps.
2. Changed `\swarmwrap@set@parshape` and `@item` patch to read `\swarmwrapremainingreg` (which Lua updates by name) instead of `\swarmwrap@remaining` (wrong register).
3. Removed hardcoded `tex.count[286]` from Lua code — only `tex.count["swarmwrapremainingreg"]` is used now.
4. `@item` patch also re-sets `\everypar` so list items continue narrowing across items.

## Turn 10 (11:00 UTC+8)

**Action**: v3.29 — ghost narrowing elimination for multi-paragraph case (self-task per Rule 0).

**Context**: Pulled latest (3fae91b, resolved merge conflict in worklog.md). Re-installed TeX Live packages (minted). All BLACKBOARD Programmer tasks marked done. Per Rule 0 ("If there is no wrapping task, INVENT ONE"), identified ghost narrowing as the most impactful remaining issue.

**Root cause analysis**: Ghost narrowing occurs when a wrapped paragraph spans a page break. TeX's `\parshape` persists to the continuation page, but the figure (`\smash{\rlap{}}`) is page-local. Previous mitigation (penalty=100000 in v3.14/v3.22) reduced but never eliminated it because: (1) penalties only affect inter-paragraph page breaks, not intra-paragraph ones (TeX bakes parshape during linebreaking), and (2) penalties were only inserted between and after narrow lines, not before the first one.

**Fix** (two parts):
1. **shipout_filter detection**: Added a `shipout_filter` callback that sets `swarmwrap_page_shipped=true` when any page is shipped. In `\swarmwrap@set@parshape` (called from `\everypar`) and in the `@item` patch, this flag is checked. If a page was shipped since `\swarmwrapnext`, the remaining counter is set to 0 and `\everypar` is cleared. This eliminates ghost narrowing for MULTI-PARAGRAPH cases (e.g., `\lipsum[1]\lipsum[2]` where the page break occurs between paragraphs).
2. **Penalty before first narrow line**: Modified `post_linebreak_filter` to also insert a penalty BEFORE the first narrow line (was only between/after). This prevents TeX from breaking just before the narrow zone starts.

**Limitation**: Single-paragraph ghost narrowing remains. When one long paragraph spans a page break, TeX's parshape is baked during linebreaking and cannot be modified by `post_linebreak_filter` or `\everypar`. The `shipout_filter` fires too late for the current paragraph. This only affects deliberately constructed stress-test scenarios; real documents with penalty=100000 rarely trigger this.
