## 20:00 UTC+8 — Task #199: Carry-over narrowing fix (v3.45)

### What was done
Implemented v3.45 of swarmwrap-callback.lua with a two-part carry-over narrowing fix:

1. **Penalty approach**: Changed `post_linebreak_filter` to insert penalty 10000 after EVERY narrow hbox (v3.44 only inserted after the LAST narrow hbox). This discourages TeX from breaking the page within the narrow wrapping zone, pushing the entire zone to the next page when space is insufficient.

2. **Ghost-narrowing shipout filter**: Added `pre_shipout_filter` callback (with `shipout_filter` fallback) that detects pages without figures (`current_page > fig_page`) and widens narrow lines to full linewidth by adjusting inter-word glue stretch. Only operates on ghost-narrowing pages to avoid text-figure overlap on normal wrapping pages.

### Key technical discovery
- `node.hpack()` was failing with "attempt to call a nil value" because the function parameter `node` shadowed the global `node` library table. Renamed to `cur` to fix.
- `node.hpack()` with 'exactly' spec caused "fuzzy token cleanup in whatsit node" fatal error when used to re-pack hbox content. Replaced with manual glue_set adjustment approach.
- Three-phase pattern detection (narrow→wide→narrow) correctly identifies carry-over but cannot distinguish it from normal wrapping on pages with figures — stretching carry-over lines on such pages causes text-figure overlap (1176 overlaps observed).

### Known limitation
Carry-over narrowing on pages that have BOTH carry-over from a previous figure AND a new figure on the same page is partially mitigated but not fully eliminated. The penalty approach reduces but cannot prevent page breaks within narrow zones when the zone exceeds remaining page space. Safe shipout modification (glue stretching) cannot distinguish carry-over from normal wrapping on these pages without risking text-figure overlap.

### Task: Fix v3.45 shipout filter — node.hpack (Task #199 follow-up)

**Result: committed as v3.46, needs QA review**

#### Root Cause
v3.45's `widen_hbox()` function set `hbox.width` and `hbox.glue_set` to stretch inter-word glue, but did NOT actually reposition text glyphs. This is the same class of bug as v3.27/v3.29 where modifying hbox metrics without re-typesetting the line only changes the reference width, not the visual output. QA verified: "Shipout filter completely fails — glue-stretch approach doesn't widen narrow lines."

#### Fix (v3.46)
1. Replaced `widen_hbox()` with `node.hpack(hbox.head, target_width_sp, 'exactly')` which actually redistributes inter-word glue and repositions all glyphs to fill the target width. This is a real re-typesetting of the line.
2. Replaced single `fig_page` counter with per-page `fig_pages` Lua table. The single counter was always equal to `current_page` in dense tests (every page has a figure), making `current_page > fig_page` always false. The table correctly tracks which pages actually have figures.
3. Kept v3.45's penalty approach (prevents carry-over at source).

#### Key Insight
The single `fig_page` counter was fundamentally flawed for dense figure layouts. Since every page had a figure, `fig_page == current_page` was always true, and the shipout filter never activated. The per-page `fig_pages` table correctly identifies pages without figures.
