## Turn 18 (15:00) — Task #208: v3.55 parshape leak regression fix

### Root Cause Analysis
The v3.54 fix for Task #205 added a correct parshape line count to `\swarmwrap@pshape` (line 1153). This made the explicit `\parshape \swarmwrap@pshape\relax` at lines 1225/1255 functional — setting N narrow + 1 full-width parshape entries for the first paragraph after `\swarmwrapnext`.

The problem: TeX's `\parshape` primitive carries remaining entries across page breaks within a paragraph. Unlike the everypar mechanism (which `buildpage_filter` can clear by setting `everypar={}`), the explicit `\parshape` entries are set at the TeX paragraph-builder level and cannot be cleared by any callback.

In v3.52, `\swarmwrap@pshape` was empty, so the explicit `\parshape` was a no-op (`\parshape \relax` = `\parshape 0`). Only everypar set parshape — clearable and safe. v3.54 inadvertently activated an un-clearable parshape source.

### Fix Applied (v3.55)
1. Removed the `\swarmwrap@pshape` build code (lines 1153-1159)
2. Removed the `\swarmwrap@pshape` rebuild in the NORMAL/SQUEEZE-FIT `\ifdim` block (lines 1241-1247)
3. Removed `\parshape \swarmwrap@pshape\relax` from both DEFERRED (line 1225) and NORMAL (line 1255) paths
4. Removed `\swarmwrap@ext@skiptrue` from both paths so everypar handles the first paragraph

All parshape is now set exclusively via the everypar extension mechanism (`\swarmwrap@apply@ext@pshape`), which `buildpage_filter` can fully clear.

## Turn 19 (16:00) — Task #211: v3.56 revert v3.55, restore wrapping

### v3.55 Failure Analysis
QA T75 found v3.55 caused **620 overlaps** (from 6 in v3.54). The everypar-only approach was fundamentally broken.

**Root cause discovered via debug tracing:** TeX's `buildpage_filter` callback fires during `\noindent` processing. This is an unpredictable TeX-internal page builder invocation that occurs even on empty pages. When it fires:
1. `buildpage_filter` reads `remaining@nl` (which was just set to nl)
2. It clears `remaining@nl = 0` and `everypar = {}`
3. When the paragraph's everypar fires, it sees `rnl=0` and does nothing
4. The first paragraph gets NO narrow parshape → full-width text → overlaps figure

**Evidence:**
```
wlog BEFORE NOINDENT: rnl=16
buildpage_filter fires → clears rnl=0, everypar={}
wlog EVERYPAR: rnl=0  ← everypar fires AFTER buildpage cleared it
```

The explicit `\parshape` (set BEFORE `\noindent`) survives the buildpage_filter because it's already loaded into TeX's paragraph builder. The everypar (which fires DURING `\noindent`) does not survive because the buildpage_filter clears it first.

### Fix Applied (v3.56)
Reverted to v3.54's explicit `\parshape` mechanism:
1. Restored `\swarmwrap@pshape` build code (lines 1153-1159)
2. Restored `\swarmwrap@ext@skiptrue` (first paragraph uses explicit parshape, not everypar)
3. Restored explicit `\parshape \swarmwrap@pshape\relax` in both DEFERRED and NORMAL paths
4. Restored pshape rebuild in NORMAL/SQUEEZE-FIT `\ifdim` block

### Key Insight
The everypar mechanism CANNOT be the sole parshape source for the first paragraph because the `buildpage_filter` fires unpredictably during `\noindent` and clears everypar state. The explicit `\parshape` is the ONLY reliable way to set parshape for the first paragraph. This means the Task #208 leak regression (42 leaks) cannot be fixed by removing the explicit parshape — it requires improving the ghost_widen/pre_shipout_filter mitigation instead.

## Turn 21 (20:00) — Task #213: v3.58 fix ghost_widen guard leak regression

### Root Cause (from QA T79 analysis)
v3.57 introduced two regressions in the parshape leak mechanism:
1. **post_linebreak_filter guard**: `if total_lines < nl and not swarmwrap_fig_on_page` prevented ghost_widen/leak_tw from being set on figure pages. When a paragraph with leftover parshape entries crossed from a figure page to a non-figure page, the continuation page had no leak_tw to widen the leaked narrow hboxes.
2. **pre_shipout_filter leak_tw clearing**: `if swarmwrap_page_had_fig then swarmwrap_leak_tw = 0` cleared leak_tw on figure pages. Even if post_linebreak_filter had set it, pre_shipout discarded it before the next non-figure page could use it.

The `nl@buf` "safety buffer" was confirmed by QA as a complete no-op (nl@buf = nl, no actual +1 line added).

### Discovery: LuaTeX cross-paragraph parshape persistence
During investigation, discovered that LuaTeX's `\parshape` persists across `\par` boundaries within the same TeX group. This caused itemize items (Test 4) to inherit narrow parshape from the first wrapped item. v3.54 masked this via pre_shipout_filter widening (because buildpage_filter cleared fig_on_page, making page_had_fig=false, so pre_shipout widened ALL narrow lines including itemize leaks).

### Fix Applied (3 changes to Lua callbacks)
1. **buildpage_filter**: Keep v3.57's `if not swarmwrap_fig_on_page` guard. This preserves Task #187's fix (don't clear rnl/everypar on figure pages).
2. **pre_shipout_filter**: Keep v3.57's page_had_fig save and fig_on_page clearing. **Key change**: remove `swarmwrap_leak_tw = 0` from the `page_had_fig=true` branch. Let leak_tw persist to the next non-figure page.
3. **fig_stack**: Clearing moved from buildpage_filter to pre_shipout_filter (v3.57 change, kept).

### Why This Works
- Figure pages: pre_shipout_filter skips widening (page_had_fig=true) but does NOT clear leak_tw → leak_tw persists
- Next non-figure page: pre_shipout_filter sees leak_tw > 0 and page_had_fig=false → widens narrow hboxes → clears leak_tw
- Task #187 preserved: buildpage_filter still skips rnl clearing on figure pages

## Turn 24 (21:00) — Task #187: Deep investigation, no safe fix found

### Problem
test-continuation-ghost.tex page 5 has 2 overlaps (y=423.5 and y=437.1,
both 113pt wide, overlapping Fig 5). These occur because the
second paragraph after Fig 5 goes full-width within the figure zone.

### Root Cause Analysis (three interacting issues)
1. **rnl exhaustion**: The first paragraph consumes most narrow parshape
   entries. post_linebreak_filter decrements rnl by narrow_count. The
   remaining rnl is insufficient for subsequent paragraphs. TeX's
   actual line spacing is slightly less than baselineskip, so the
   first paragraph can consume nl narrow entries before the figure
   visually ends.
2. **Page-space clipping**: The v3.40 everypar clipping reduces rnl to
   fit remaining page space (pagegoal - pagetotal - 3*baselineskip).
   On figure pages where the figure is mid-page, this clips rnl to
   the page bottom, not the figure bottom — prematurely exhausting
   rnl for subsequent paragraphs within the figure zone.
3. **buildpage_filter rnl zeroing**: After a figure page ships,
   the next page's buildpage_filter sees fig_on_page=false and zeros
   rnl, preventing continuation-page everypar from providing
   narrow parshape for paragraphs that started on the figure page.

### Fixes Attempted (all reverted due to regressions)
1. **nl_buf = nl + 1** (parshape safety buffer): Adds 1 extra narrow
   parshape entry. Prevents overlap from parshape exhaustion.
   But first paragraph's narrow_count = nl_buf while rnl = nl, so
   rnl goes to 0. Setting rnl = nl_buf caused ghost-narrowing
   regressions (+1 page on test-ghost-narrowing, 13 overlaps from 0).
2. **fig_on_page guard + fig_recent**: In post_linebreak_filter, only
   trigger ghost_widen+rnl zeroing on non-figure pages. Combined
   with nl_buf and fig_recent (extends fig_on_page to next page),
   this reduced overlaps 2→0 but caused massive regressions:
   ghost-narrowing 0→13 overlaps, pagebreak-variations 0→2 overlaps.
3. **first_para_done flag**: Skip rnl decrement for the first
   paragraph (handled by explicit parshape). Preserves full nl_buf
   for subsequent paragraphs. Combined with figure-page clipping
   (-1bs instead of -3bs), reduced overlaps 2→0 but caused
   ghost-narrowing +1 page, 13 overlaps.

### Conclusion
The remaining overlap is a known limitation of the everypar mechanism
(per Task #144 — fundamental TeX behavior). Eliminating it without
regressions requires a fundamental redesign of the rnl tracking
system — e.g., computing per-paragraph rnl based on actual figure
coverage at each paragraph start rather than decrementing a global
counter. This is beyond safe incremental changes. No code changes
committed. v3.58 remains current version.

## Turn 25 (22:00) — Task #214: v3.59 fix fig_stack timing regression

### Root Cause
QA T81 (in TinyTeX) found v3.58 causes 45 overlaps in stress-50
(was 4 in v3.54). Root cause: v3.57/v3.58 moved
`swarmwrap_fig_stack = {}` from `buildpage_filter` (fires at each
page-break decision point) to `pre_shipout_filter` (fires only
at page ship). Between buildpage calls on the same page, stale
fig_stack entries corrupt post_linebreak_filter's remaining_nl
calculation, producing wrong parshape for subsequent paragraphs.

### Fix Applied (v3.59)
Moved `swarmwrap_fig_stack = {}` back to `buildpage_filter`
(v3.54 timing). Kept `fig_on_page = false` in `pre_shipout_filter`
(v3.57/v3.58 timing) so Task #187's `not fig_on_page` guard in
buildpage works correctly across multiple buildpage calls on the
same figure page. Kept v3.58's leak_tw preservation on figure
pages (don't clear leak_tw when page_had_fig=true).

### Design Rationale
The fix splits the v3.57/v3.58 changes:
- **fig_on_page**: Stays in pre_shipout (v3.57 timing). Must persist
  across all buildpage calls on a figure page so the guard
  `if not fig_on_page` correctly identifies figure pages.
- **fig_stack**: Back in buildpage (v3.54 timing). Must be clean at
  each buildpage call because post_linebreak_filter reads it
  between calls. Stale entries cause wrong remaining_nl.
- **leak_tw**: Stays in pre_shipout with figure-page preservation
  (v3.58 timing). On figure pages, leak_tw persists to the next
  non-figure page for widening.
