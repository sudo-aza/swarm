## 14:00 UTC+8 — v3.39: Reduce excessive narrowing (Task #185)

### What was done
Split figure height into two separate values to decouple parshape computation from overlap prevention:
1. `fh_narrow` = ht+dp (no padding) — used for parshape nl count, \par patch extension, and DEFERRED fit check
2. `fh_safe` = ht+dp+4*bs (full padding) — used ONLY for vspace push (cross-session overlap prevention)

### Root cause
The 4*bs padding in v3.38 was added to prevent hollow carry-over, but it was used for BOTH parshape line count AND overlap prevention. This caused the narrowing zone to extend 3.5x beyond the actual figure height on average (worst case 10.1x). The fix separates these concerns: the narrowing zone now closely matches the figure height, while overlap prevention is maintained via the vspace mechanism.

- `\swarmwrapnext`: uses fh@narrow@val for nl computation, fig_bottom (narrow) for \par patch, fh@val (full) for fig_bottom@safe and vspace push
- DEFERRED fit check uses fh@narrow@val (fewer unnecessary page ejects)

## 20:00 UTC+8 — v3.42: Visible content height via Lua (Task #190)

### What was done
Implemented Lua node traversal to measure actual visible figure height (max rule node height) for parshape line count, replacing `\ht+\dp` box height. The box height includes minipage overhead (struts, parskip, aboveskip, depth padding) that does not correspond to visible figure content, inflating the parshape line count.

### Root cause analysis
The `\ht+\dp` box height for a 2cm figure is 119pt (ht=57pt + dp=62pt), but the visible colored rectangle (rule node) is only 57pt. The remaining 62pt is minipage internal spacing. Using 119pt for parshape gives nl=9 lines (122pt zone), when only 5 lines (68pt) are needed. This 2.1x overestimation causes the narrowing zone to extend far beyond the figure.

### Changes to swarmwrap-callback.lua (v3.42)
- Added `find_max_rule_height()`: recursively traverses node list to find max rule height
- Added `swarmwrap_measure_visible_height()`: measures visible content height
- v3.42 refinement: removed 0.5*bs bonus and 2*bs minimum (v3.41 additions that crossed baselineskip boundaries for 6cm/7cm figures)
- Uses raw max_rule for fh_narrow — parshape loop naturally rounds up (ceil)

### Known limitation: detection metric aggregation
The detection script's excessive narrowing metric computes `narrow_zone / fig_height` where narrow_zone is the Y extent of ALL narrow lines on the page to the LEFT of the figure. On pages with multiple figure sessions, narrow lines from ALL sessions are aggregated. Additionally, stray narrow text (section numbers, short text lines from other paragraphs) inflates the zone. For example, on page 2 (single 4cm figure): the actual wrapping zone is ~108pt (0.96x ratio), but stray narrow text extends the measured zone to 585pt (5.16x ratio). This is a detection metric design limitation, not a parshape accuracy issue.

## 22:00 UTC+8 — v3.43: Fix caption-text overlap (Task #196)

### What was done
Changed visible content height measurement in swarmwrap-callback.lua from `find_max_rule_height()` (which only measured the colored rectangle/rule node height) to `box.height + box.depth - strutbox.height - strutbox.depth` (full box height minus strut overhead). This covers the full figure+caption block instead of just the colored rectangle.

### Root cause
v3.42's Lua `find_max_rule_height()` traversed the box node tree and returned only the maximum rule node height (the colored rectangle). For a 4cm figure with caption, the rule is 113.8pt but the full box (including caption text) is ~137.4pt. The 24pt difference (caption height) was NOT covered by the parshape, so body text at full width ran beside the caption on ALL 50 figures. With v3.43, the parshape zone covers the full figure+caption height, keeping body text narrow throughout.

### Changes to swarmwrap-callback.lua (v3.43)
- Removed `find_visible_extent()` function (attempted node-tree y-coordinate tracking — caused TeX stack overflow due to compounding vlist offsets)
- Kept `find_max_rule_height()` as fallback (still used for debug logging)
- Simplified `swarmwrap_measure_visible_height()` to use `box.height + box.depth - strutbox.height - strutbox.depth`
- Added minimum of 1*bs (13.6pt) to prevent zero/negative values
- Debug logging now shows: box ht, dp, max_rule, and final visible height

### Changes to swarmwrap.sty (v3.43)
- Version bump: v3.42 → v3.43
- Updated wlog message to say "visible content height incl caption"

## 23:00 UTC+8 — v3.44: Use raw box height for visible content (Task #190 refinement)

### What was done
Fixed v3.43's caption-text overlap regression by removing the strut subtraction from
the visible content height measurement. v3.43 subtracted `strutbox.height + strutbox.depth`
(~13.6pt for LM 11pt) from `box.height + box.depth`. Debug node traversal revealed
this was WRONG: for `[t]`-minipages with tall figures (rule > strut), `box.height` already
equals the rule height (the strut is hidden behind the taller rule). Subtracting strut_h
removed actual caption content instead of invisible strut overhead, causing body text
to overlap captions.

### Root cause analysis
Debug walk of the box node structure for a 3cm figure in [t]-minipage:
```
VLIST ht=85.36 dp=37.20 wd=85.36
  HLIST ht=85.36 dp=0.00  ← rule line (figure)
  GLUE wd=10.00            ← \abovecaptionskip
  GLUE wd=0.00              ← inter-line glue
  GLUE wd=0.00              ← another inter-line glue
  HLIST ht=23.12 dp=4.08   ← caption line
  GLUE wd=0.00              ← trailing glue
```
box.height = 85.36pt (the rule, NOT the strut — strut is shorter)
box.depth = 37.20pt (abovecaptionskip + caption)
strut_h = 13.60pt (LM 11pt)
v3.43: visible = 85.36 + 37.20 - 13.60 = 108.96pt → nl=8 lines = 108.8pt
Caption bottom: 95.36 + 23.12 + 4.08 = 122.56pt → 108.8 < 122.56 → OVERLAP!

v3.44: visible = 85.36 + 37.20 = 122.56pt → nl=10 lines = 136.0pt > 122.56 → OK ✓
