**Critical Bug Found — Non-Deterministic Word Extraction**:

The `TexExtractor._preprocess()` method builds a regex alternation from `LITERAL_ENVS | MATH_ENVS` (set union). Python's `PYTHONHASHSEED` randomization changes the set iteration order. When `code` (4 chars) appears before `codeblock` (9 chars) in the alternation, Python's regex engine matches `\begin{code` instead of `\begin{codeblock}` due to the prefix-match behavior with 27 alternatives. The scanner then searches for `\end{code}` (which doesn't exist), silently skipping ALL remaining content in the file.

**Reproduction**:
```
PYTHONHASHSEED=0 python3 scripts/spellcheck.py demo-beautiful.tex → 463 words (WRONG — 131 words missing)
PYTHONHASHSEED=2 python3 scripts/spellcheck.py demo-beautiful.tex → 594 words (correct)
```

Tested 8 seeds: 0, 1, 999 give 463 (wrong); 2, 3, 42, 100, 12345 give 594 (correct). ~37.5% failure rate.

**Root Cause**: The regex `re.match(r"^\s*\\begin\{(code|codeblock|...25 more...)\}")` with 27 alternatives where `code` is a prefix of `codeblock`. When `code` is tried first and matches, the engine finds `code\}` doesn't match (next char is `b`), but doesn't always backtrack to try `codeblock` with this many alternatives.

**Other False Positives**: tabularray syntax (`font=`, `bg=sbLight`, `1pt`) and number+symbol tokens (`20+`) leak as words.

**Rating: FAIL (8/10)** — Critical non-determinism makes the tool unreliable. Fix tasks #85 (critical: sort alternation by length), #86 (moderate: multi-line math), #87 (minor: tabularray syntax) created for Programmer.
