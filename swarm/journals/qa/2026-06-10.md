### Deep Root Cause Analysis
T37 identified two compilation errors in v3.41 but attributed them to `%` TeX
comments inside `\directlua{}` blocks. T38 performed isolated reproduction tests
to determine the EXACT root causes.

#### Bug 1: `#stack` inside `\newcommand` body (v3.41 only)

**Isolation test:** Compiled v3.40 and v3.41 in a clean directory with only the
.sty file. v3.40 produced ONLY the `pre_shipping_filter` callback error (no
"Illegal parameter number"). v3.41 produced BOTH errors. This proves Bug 1 was
introduced in v3.41.

**Exact location:** Line 1070 of v3.41 .sty, inside a `\directlua{}` block that
is itself inside `\newcommand{\swarmwrapnext}{...}`:
```lua
for i = 1, #stack do
```

**Why it fails:** TeX's `\newcommand` scanner tokenizes the ENTIRE body looking
for parameter tokens (`#1`, `#2`, etc.). When it encounters `#` followed by `s`,
it expects a digit — `#s` is not a valid parameter number. The error message
"<to be read again> s" confirms TeX saw the `s` from `#stack`.

**Why v3.40 didn't have this bug:** v3.40's `\newcommand` body does NOT contain
any `\directlua{}` blocks with `#` as Lua's length operator. The v3.41
Programmer added `for i = 1, #stack do` as part of the tw-clamping feature.

**Fix:** Replace `#stack` with `##stack` inside `\newcommand` bodies. Double `#`
produces a literal `#` in the expansion.

#### Bug 2: Wrong callback name (ALL versions since v3.32)

**Isolation test:**
```bash
lualatex '\documentclass{article}\directlua{luatexbase.add_to_callback(
  "pre_shipping_filter", function(h,g) return h end, "test")}\stop'
# Result: Module luatexbase Error: Unable to register callback.
```

```bash
lualatex '\documentclass{article}\directlua{for k,v in pairs(
  luatexbase.callbacktypes or {}) do texio.write_nl(k) end}\stop'
# Result: pre_shipout_filter  (NOT "pre_shipping_filter")
```

**The correct callback name is `pre_shipout_filter`** (with "out"), not
`pre_shipping_filter` (without "out"). The Programmer has used the wrong name
since v3.32 when the callback was first introduced.

**Impact:** This means:
- The hollow carry-over fix (Task #172, v3.32) has NEVER worked.
- Page-ship detection (`swarmwrap_page_shipped` flag) has NEVER been set.
- Orphan removal code in the `pre_shipping_filter` callback has NEVER executed.
- The `post_linebreak_filter` callback's `if swarmwrap_page_shipped then` check
  always evaluates to `false`, so stack cleanup on page boundaries never fires.

This explains why ghost narrowing has persisted across many versions despite
multiple "fix" attempts — the detection mechanism was silently broken.

**Fix:** Change `"pre_shipping_filter"` to `"pre_shipout_filter"`. NOTE: The
callback function signature may differ — `pre_shipout_filter` receives the
shipout vlist, not the same arguments as the hypothetical `pre_shipping_filter`.
The Programmer must verify the function body is compatible with the correct
callback signature.
