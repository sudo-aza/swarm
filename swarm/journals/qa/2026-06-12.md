### Step 4.5 / Rule 5: New detection script

T46 used `analyze-wrapping.py` which has known false-positive issues for ghost
narrowing detection. The script flags ANY line narrower than 300pt on a
figure-less page as "ghost," but this includes:
- Page numbers (e.g., "2" at w=5.5pt)
- Short paragraphs that broke at a page boundary (e.g., a single line at
  w=163pt that is just the tail end of a paragraph)
- Legitimate continuation text that happens to be in a narrow parshape zone

Created `scripts/detect-parshape-leak.py` (276 lines) that detects the
DEFINITIVE parshape leak signature: **fragmented lines** — multiple text
spans at the same Y coordinate on a figure-less page. This pattern can ONLY
occur when TeX's parshape is active with multiple indent/width segments,
which should never happen on a page with no figure.

### Script design

Key features:
1. **Fragment detection**: Groups text spans by Y coordinate (0.5pt tolerance).
   If >=2 spans share the same Y, the line is "fragmented" — definitive proof
   of active parshape.
2. **Leak source tracking**: For each leaked page, reports the figure
   rectangles from the PREVIOUS page, establishing the leak origin.
3. **Narrow zone detection**: Also captures narrow (non-fragmented) lines that
   appear between fragmented lines within the same "leak zone."
4. **Severity classification**: none/mild/moderate/severe based on fragment
   count and total leaked lines per page.
5. **False positive elimination**: Pages with short single-fragment narrow
   lines (e.g., page 2 of customwrap with just a paragraph remnant and page
   number) are correctly classified as "clean."

### Results with new script (v3.47)

| Test | Old ghost count | New frag lines | Leaked pages | Severity |
|------|----------------|----------------|--------------|----------|
| stress-50 | 0 | 0 | 0 | clean |
| customwrap | 42 | 6 | 2 | mild+moderate |
| ghost-narrowing | 36 | 10 | 3 | mild+2x moderate |
| pagebreak-variations | 77 | 21 | 7 | all moderate |
| continuation-ghost | 5 | 1 | 1 | mild |
| **TOTAL** | **160** | **38** | **13** | |

The 160→38 reduction (76%) confirms most "ghost" lines from the old script
were false positives. The 38 remaining fragmented lines are the TRUE parshape
leak count — lines that are visually broken into multiple fragments at
different X positions due to leaked parshape segments.

### Key insight: Root cause refinement

All 13 leaked pages show the same pattern:
1. Previous page had a figure (leak source identified)
2. The page shipped with the figure still active
3. Hboxes already had parshape applied by `post_linebreak_filter`
4. `buildpage_filter` resets the Lua figure stack
5. But the already-typeset hboxes RETAIN their parshape and ship to the new page

This means `buildpage_filter` alone cannot fix the leak — it resets Lua state
but cannot retroactively remove parshape from typeset material. The fix must
happen EITHER in `post_linebreak_filter` (detect imminent page ship and widen
hboxes) OR in `buildpage_filter` (modify the vlist to replace narrowed hboxes
at the top of the new page).

### Verified false positive elimination

Customwrap page 2: old script flagged 2 "ghost" lines (w=163.2pt "Pellentesque
cursus luctus mauris." and w=5.5pt "2" page number). New script correctly
identifies this as CLEAN — the 163pt line is a single-fragment short paragraph
that broke at the page boundary, not a parshape leak.
